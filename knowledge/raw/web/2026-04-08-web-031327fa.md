# Harness Permission Pipeline: 5-Layer Defense-in-Depth

**Date**: 2026-03-31
**Source**: `claude-harness-src.zip` — `src/utils/permissions/`, `src/Tool.ts`, `src/hooks/toolPermission/`
**Task**: [HARNESS_PERMISSION_PIPELINE]

## Executive Summary

The Claude Code harness implements a 5-layer permission pipeline that evaluates every tool invocation through successive gates before execution. The design is defense-in-depth: each layer can deny or escalate, and the final decision is the most restrictive intersection of all layers. The system supports 6 permission modes (default, plan, acceptEdits, bypassPermissions, dontAsk, auto) and 3 rule behaviors (allow, deny, ask).

## The 5 Layers in Order

### Layer 1: Zod Schema Validation (`tool.inputSchema.parse`)
- Every tool defines a Zod v4 schema for its inputs
- `hasPermissionsToUseToolInner` at line 1215: `tool.inputSchema.parse(input)`
- Malformed input is rejected before any permission logic runs
- If parse fails, the tool falls through with a `passthrough` behavior (default ask)
- **Key insight**: This is a structural firewall — no tool can receive inputs outside its declared schema

### Layer 2: Rule Matching (deny → ask → tool.checkPermissions → bypass → allow)
This is the core decision engine in `hasPermissionsToUseToolInner` (lines 1158-1319):

**Step 1a — Deny rules**: `getDenyRuleForTool()` checks if the entire tool is denied. Immediate reject.

**Step 1b — Ask rules**: `getAskRuleForTool()` checks if the tool requires explicit confirmation. Exception: sandboxed Bash commands can skip this.

**Step 1c — Tool-specific permission check**: `tool.checkPermissions(parsedInput, context)` — each tool implements its own permission logic:
- Bash: parses compound commands, checks each sub-command against prefix/wildcard/exact rules
- File tools: check path against working directory, detect sensitive paths (.git/, .claude/, shell configs)
- Agent: checks agent type against deny rules

**Step 1d-1g — Safety escalations**: Even in bypass mode, these force a prompt:
- Tool-level deny (1d)
- `requiresUserInteraction()` tools (1e)
- Content-specific ask rules with `ruleBehavior:'ask'` (1f)
- Safety checks for sensitive paths — `.git/`, `.claude/`, `.vscode/`, shell configs (1g) — bypass-immune

**Step 2a — Mode check**: If `bypassPermissions` mode, allow (unless safety-escalated above)

**Step 2b — Tool-level allow rules**: `toolAlwaysAllowedRule()` — entire tool whitelisted

**Step 3 — Passthrough → ask**: Any remaining `passthrough` result becomes `ask`

### Layer 3: Hook System (`executePermissionRequestHooks`)
- Runs `PermissionRequest` lifecycle hooks registered in settings.json
- Hooks can: allow (with optional permission updates), deny (with optional interrupt), or pass through
- For headless agents: hooks run via `runPermissionRequestHooksForHeadlessAgent()` — the only chance to allow before auto-deny
- Hooks can modify input and persist new permission rules

### Layer 4: Classifier (Auto Mode / YOLO Classifier)
When in `auto` mode (or `plan` mode with auto enabled):

**Fast paths** (skip classifier API call):
1. `acceptEdits` simulation — re-run `tool.checkPermissions` with mode=acceptEdits. If it would allow, skip classifier
2. Safe tool allowlist (`SAFE_YOLO_ALLOWLISTED_TOOLS`) — ~25 read-only/metadata tools auto-allowed
3. Dangerous pattern stripping — rules like `Bash(python:*)` are stripped at auto-mode entry to prevent classifier bypass

**Classifier invocation** (`classifyYoloAction`):
- Uses a **sideQuery** — a separate Claude API call with the full conversation transcript + the pending action
- Two-stage evaluation (stage1 + stage2 for complex cases)
- System prompt includes user-customizable allow/soft_deny/environment rules
- Model: configurable, uses `sideQuery()` utility
- Result: `shouldBlock` boolean + `reason` string + token usage metrics

**Fail-closed behavior** (`tengu_iron_gate_closed` gate):
- If classifier API errors: deny by default (fail closed)
- Feature gate can switch to fail-open for degraded operation
- Transcript-too-long: falls back to manual prompting (not classifier retry)

**Denial tracking** (`denialTracking.ts`):
- Tracks consecutive denials (limit: 3) and total denials (limit: 20)
- After 3 consecutive classifier denials: falls back to interactive prompting
- After 20 total: same fallback
- Any successful tool use resets consecutive counter

### Layer 5: Interactive Prompt / Multi-Agent Delegation
Three permission handlers based on context:

1. **Interactive handler** (`interactiveHandler.ts`): Main agent — pushes to React UI queue, races classifier check against user interaction. User can: approve (once or always), reject, modify input, provide feedback.

2. **Coordinator handler** (`coordinatorHandler.ts`): For coordinator workers — awaits hooks + classifier sequentially, then falls through to interactive dialog.

3. **Swarm worker handler** (`swarmWorkerHandler.ts`): For swarm workers — tries classifier first, then forwards permission request to leader via mailbox. Leader approves/denies via the interactive handler. Abort signal cleanup prevents hanging.

## Permission Rule Architecture

### Rule Sources (priority/trust hierarchy)
```
policySettings     — Managed/enterprise (highest trust)
flagSettings       — Feature flag controlled
userSettings       — ~/.claude/settings.json (user-global)
projectSettings    — .claude/settings.json (repo-level)
localSettings      — .claude/settings.local.json (gitignored)
cliArg             — --allowedTools CLI argument
command            — /allowed-tools slash command
session            — In-memory, per-session
```

### Rule Types
```
PermissionRuleValue = { toolName: string, ruleContent?: string }
Examples:
  "Bash"                  — entire tool
  "Bash(npm test:*)"      — prefix: npm test + any args
  "Bash(git commit *)"    — wildcard: git commit with any flags
  "Bash(npm publish)"     — exact command
  "mcp__server1"          — all tools from MCP server
  "Agent(Explore)"        — specific agent type
```

### Rule Behaviors
- `allow` — auto-permit without prompting
- `deny` — reject, model receives denial message
- `ask` — force interactive prompt even in auto/bypass mode

### Managed Policy (`allowManagedPermissionRulesOnly`)
When enterprise policy sets this flag:
- Only `policySettings` rules are respected
- User/project/local rules are ignored
- "Always allow" options hidden from UI
- New rules cannot be persisted

## Dangerous Pattern Protection

The harness maintains explicit lists of dangerous command patterns that cannot be allowlisted in auto mode:

**Cross-platform code execution** (23 entries): python, node, deno, ruby, perl, php, lua, npx, bunx, npm/yarn/pnpm/bun run, bash, sh, ssh

**Bash-specific** (additional): zsh, fish, eval, exec, env, xargs, sudo

**Ant-internal** (additional): gh, gh api, curl, wget, git, kubectl, aws, gcloud, gsutil

`isDangerousBashPermission()` catches: tool-level allow, prefix rules for interpreters, wildcard rules matching interpreters.

## Bypass Killswitch

- `bypassPermissionsKillswitch.ts` — can remotely disable `--dangerously-skip-permissions` via Statsig gate
- Runs once before first query
- Also validates auto mode gate access (model support, org permissions)
- `/login` resets the check to re-evaluate with new org context

## Key Design Patterns for Clarvis

### 1. Layered Evaluation with Early Exit
Each layer can deny or escalate. Allow only if ALL layers agree. The order matters:
- Structural validation first (cheap, no I/O)
- Static rules second (fast lookup)
- Tool-specific logic third (may read filesystem)
- External classification fourth (expensive API call)
- Human-in-the-loop last (most expensive)

### 2. Safety Checks Are Bypass-Immune
Even `--dangerously-skip-permissions` cannot override:
- Deny rules (always respected)
- Content-specific ask rules
- Safety checks for sensitive paths
- `requiresUserInteraction()` tools

### 3. Graceful Degradation
- Classifier unavailable → fail closed (deny) by default
- Classifier transcript too long → fall back to manual
- Hook failures → fall through to next layer
- Denial limits → fall back to prompting

### 4. Multi-Agent Permission Delegation
Three distinct handlers for three execution contexts:
- Direct interactive: UI prompt
- Coordinator worker: automated checks → UI prompt
- Swarm worker: classifier → forward to leader → leader decides

## Clarvis Design Doc: Tool Permission System

### Proposed Architecture

Replace `--dangerously-skip-permissions` with a 4-layer permission system:

```
Layer 1: Schema Validation (Zod/JSON Schema)
  └─ Validate tool input structure before any logic

Layer 2: Static Rule Engine
  └─ YAML-based rules in workspace/config/permissions.yaml
  └─ 3 behaviors: allow, deny, ask
  └─ Rule scopes: tool-level, content-prefix, wildcard
  └─ Sources: system (Clarvis defaults), user, project, session

Layer 3: Hook Pipeline
  └─ Pre-tool hooks that can allow/deny/modify
  └─ Cost gate: deny if budget exceeded
  └─ Rate limiter: deny if too many tool calls per minute
  └─ Path guard: deny writes outside workspace/

Layer 4: Decision Handler (context-dependent)
  └─ Autonomous mode: deny by default (safest)
  └─ User-facing mode: queue for Telegram approval
  └─ Multi-agent mode: forward to orchestrator (Clarvis)
```

### Implementation Plan

**Phase 1 (M effort): Permission Rule Engine**
- `scripts/permission_engine.py` — rule loading, matching, evaluation
- Config: `workspace/config/permissions.yaml`
- Default rules: allow read-only brain operations, deny file deletion, ask for external API calls
- Integration: wrap `spawn_claude.sh` to inject `--allowedTools` based on rules

**Phase 2 (M effort): Hook Pipeline**
- Pre-tool hooks: cost gate, rate limiter, path guard
- Post-tool hooks: cost logging, audit trail
- Config: hook definitions in permissions.yaml

**Phase 3 (L effort): Multi-Agent Permission Delegation**
- Project agents forward permission requests to Clarvis orchestrator
- Clarvis evaluates rules + can escalate to Telegram for user approval
- Timeout-based auto-deny for unattended operation

**Phase 4 (S effort): Dangerous Pattern Protection**
- Port DANGEROUS_BASH_PATTERNS list
- Prevent allowlisting of interpreter prefixes
- Path safety checks for .git/, .claude/, config files

### Key Differences from Harness

| Aspect | Harness | Clarvis Proposed |
|--------|---------|-----------------|
| Runtime | Single-process TypeScript | Multi-process Python + Claude Code |
| UI | React terminal / bridge | Telegram bot + CLI |
| Classifier | LLM sideQuery (expensive) | Rule-based only (no LLM overhead) |
| Trust model | User controls everything | Dual: user + autonomous agent |
| State | In-memory + JSON settings | YAML config + brain metadata |
| Multi-agent | Swarm mailbox | Orchestrator pattern (project_agent.py) |

### Migration Path
1. Start with `permissions.yaml` for new project agents (don't break existing cron)
2. Cron jobs keep `--dangerously-skip-permissions` until rule engine is proven
3. User-facing Telegram interactions get permission prompts first
4. Eventually retire `--dangerously-skip-permissions` for all non-cron spawns
