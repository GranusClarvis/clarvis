## Overview

Hermes Agent is NousResearch's fully open-source AI agent framework (MIT license, Python, ~88MB repo). It describes itself as "an autonomous agent that lives on your server, remembers what it learns, and gets more capable the longer it runs." It connects across CLI, Telegram, Discord, Slack, and WhatsApp, with sandboxed execution via local, Docker, SSH, Singularity, or Modal backends.

**License: MIT** (confirmed in pyproject.toml: `license = "MIT"`, version 0.1.0, author Nous Research). No LICENSE file was found at the standard path; the GitHub API also reports `license: null`, but the pyproject.toml and README badge both declare MIT.

## 1. State Management & Persistence

### 1.1 SQLite Session Store (`hermes_state.py`)

The central persistence layer is a **SQLite database** at `~/.hermes/state.db` using WAL mode for concurrent access (multiple reader threads, single writer). It replaces per-session JSONL files.

**Schema (two tables + FTS5 virtual table):**

- **sessions table**: `id` (TEXT PK), `source` (cli/telegram/discord/etc.), `model`, `model_config` (JSON), `system_prompt`, `user_id`, `started_at`, `ended_at`, `end_reason`, `message_count`, `tool_call_count`, `input_tokens`, `output_tokens`, `parent_session_id` (for compression-triggered splits).
- **messages table**: `role`, `content`, `timestamp`, `tool_name`, `tool_calls` (JSON), `tool_call_id`, `token_count`, `finish_reason`. FK to sessions.
- **FTS5 virtual table**: Maintained by triggers, enables full-text search with phrases, Boolean operators, and prefix matching across all session messages.

**Key operations**: `create_session()`, `end_session()`, `append_message()`, `get_messages_as_conversation()`, `search_messages()` (FTS5), `export_session()`, `export_all()` (JSONL backup), `prune_sessions()` (time-based cleanup).

**Schema versioning**: Tracked in a dedicated table; migration logic included (v1 to v2 adds `finish_reason` column).

**Thread safety**: `check_same_thread=False`, WAL journal mode, FK constraints enabled, Row factory for dict-like access.

### 1.2 Session Lifecycle (`gateway/session.py`)

- **SessionSource**: Origin metadata (platform, chat ID, user, thread).
- **SessionContext**: Complete session info including connected platforms, home channels, timestamps.
- **SessionEntry**: Maps session keys to session IDs with creation/update times, token counts, display names, auto-reset flags.
- **Session generation & routing**: `_generate_session_key()` creates unique IDs from source details; `get_or_create_session()` evaluates reset policies.
- **Reset policies**: `_should_reset()` checks idle timeouts and daily reset schedules, respecting active background processes.
- **Dual persistence**: SQLite (via `SessionDB`) with JSONL fallback when SQLite unavailable.
- **Transcript logging**: `append_to_transcript()` writes to both database and legacy files; `load_transcript()` prioritizes database.

### 1.3 Background Process Registry

`tools/process_registry.py` manages background processes, checkpointing to `~/.hermes/processes.json`.

### 1.4 Configuration Persistence (`~/.hermes/`)

```
~/.hermes/
  config.yaml        — Settings with version tracking for migrations
  .env               — API keys (python-dotenv)
  SOUL.md            — Global personality/persona
  memories/          — MEMORY.md + USER.md
  skills/            — Agent-authored skill documents
  sessions/          — Gateway session data
  cron/              — Scheduled job configs + lock files
  logs/              — Operational logs
  pairing/           — DM authorization codes
  hooks/             — Event handler directories
  processes.json     — Background process state
  state.db           — SQLite session database
```

**Config migrations**: Version bumps in `_config_version` trigger interactive prompts for new required fields.

## 2. Memory System

### 2.1 Bounded Persistent Memory (`tools/memory_tool.py`)

Two markdown files serve as persistent memory:

- **MEMORY.md** (~2,200 character limit, ~800 tokens): Agent's personal notes — environment facts, conventions, lessons learned.
- **USER.md** (~1,375 character limit, ~500 tokens): User profile — preferences, communication style, habits.

**Frozen Snapshot Pattern**: System prompt receives a static memory state at session start. The underlying files update immediately on disk, but the prompt snapshot remains frozen — this **preserves the prefix cache** throughout the conversation while maintaining durable persistence.

**Atomic writes**: Uses temp-file + rename pattern (not `open("w")` + `flock`) to prevent the race where `"w"` truncates before lock acquisition.

**Entry management**: Section sign (`§`) delimiter separates entries. Exact deduplication at load. Substring matching for replace/remove operations (rejects if multiple distinct entries match).

**Learning triggers** (when the agent decides to save):
- User shares preferences or personal details
- Environment discovery (OS, tools, project structure)
- Explicit corrections ("remember this")
- Learned conventions, API quirks, workflow specifics
- Task completion diary entries
- Solved problems or discovered techniques

### 2.2 Session Search (`tools/session_search_tool.py`)

Enables **cross-session memory** by searching past session transcripts via FTS5, then summarizing matching sessions using an auxiliary model (Gemini Flash). Features:
- Query with optional role filter, up to 5 sessions
- Parent-session resolution for delegated conversations
- Truncation around query matches (~100k chars)
- Async summarization with retry logic (3 attempts, exponential backoff)

### 2.3 Context Compression (`agent/context_compressor.py`)

Manages conversation memory by summarizing older dialogue turns when approaching model token limits.

**Strategy**: "Protect first N + last N turns, summarize everything in between."
- Threshold: 85% of context capacity (configurable)
- Protects first 3 turns + last 4 turns
- Summary target: ~500 tokens, generated by auxiliary model (Gemini Flash)
- Summaries prefixed with `[CONTEXT SUMMARY]:`, emphasize tool invocations, retrieved info, key decisions
- Fallback: Simple truncation when no auxiliary client available
- Creates session splits in SQLite with parent references after compression

## 3. Self-Improvement Mechanisms

### 3.1 Skill Generation (`tools/skill_manager_tool.py`)

The primary self-improvement mechanism: **the agent writes reusable procedural knowledge after solving hard problems**.

**Storage**: `~/.hermes/skills/<skill-name>/SKILL.md` with optional `references/`, `templates/`, `scripts/`, `assets/` subdirectories.

**Six actions**: create, edit, patch (find-and-replace), delete, write_file, remove_file.

**Validation**: Name pattern `^[a-z0-9][a-z0-9._-]*$` (max 64 chars). Required YAML frontmatter with `name` and `description`. Description capped at 1,024 chars. Path traversal blocked.

**Skill creation triggers**:
- Complex tasks requiring 5+ function calls
- Overcoming errors with durable solutions
- User-corrected approaches proving effective
- Non-trivial workflow discoveries
- Explicit user requests to remember procedures

**Patch-driven updates** enable incremental improvement without full rewrites. Error recovery hints provide file previews on failed patches for self-correction.

### 3.2 Skills Hub (`tools/skills_hub.py`)

Community skill discovery and installation from **four registries**:
1. **GitHubSource** — User-configured repos ("taps"), defaults include `openai/skills`, `anthropics/skills`, `VoltAgent/awesome-agent-skills`
2. **ClawHubSource** — clawhub.ai marketplace (note: "ClawHavoc incident showed their vetting is insufficient — 341 malicious skills found Feb 2026")
3. **ClaudeMarketplaceSource** — Claude Code plugin marketplace repos
4. **LobeHubSource** — 14,500+ LobeHub agents converted to SKILL.md format

**Trust levels**: builtin, trusted, community. Security scanning via `tools.skills_guard`. Quarantine workflow: skills staged in `.hub/quarantine` before install. Audit logging for all installations.

### 3.3 Progressive System Prompt Assembly (`agent/prompt_builder.py`)

The system prompt dynamically assembles multiple layers:
- Default agent identity
- Memory guidance (instructs agent to proactively save learnings)
- Session search guidance (reference past conversations)
- Skills guidance (save approaches as reusable skills after complex tasks)
- Platform hints (per-platform markdown rendering)
- Skills index (scanned from `~/.hermes/skills/`, grouped by category)
- Context files (AGENTS.md, .cursorrules, SOUL.md)
- Frozen memory snapshot

Skills index uses progressive disclosure: only descriptions loaded into prompt; full skill content loaded on-demand via `skill_view()`.

### 3.4 RL Training Infrastructure (`tools/rl_training_tool.py`)

Full reinforcement learning training management for the **Tinker-Atropos** framework, enabling the agent to orchestrate its own fine-tuning:

**Environment discovery**: AST-based scanning for `BaseEnv` subclasses without importing. Dynamic introspection of Pydantic config fields.

**Training lifecycle**: Spawns three concurrent processes:
1. Atropos API (trajectory storage)
2. Tinker trainer + inference (LoRA rank 32, lr 4e-5, checkpoints every 25 steps)
3. Environment server

**Locked infrastructure parameters**: Tokenizer Qwen/Qwen3-8B, max 8192 tokens, up to 2048 workers, 2500 training steps.

**Inference testing**: Validates environments pre-training using three models across intelligence scales (48 rollouts per model, ~144 total).

**Monitoring**: Rate-limited WandB metrics retrieval. Full run state tracking (pending/starting/running/stopping/stopped/completed/failed).

### 3.5 Trajectory Saving & Compression

**Trajectory saving** (`agent/trajectory.py`): Appends to JSONL files in ShareGPT format. Successful runs go to `trajectory_samples.jsonl`, failed to `failed_trajectories.jsonl`. Each entry: conversations, timestamp, model, completed flag.

**Trajectory compression** (`trajectory_compressor.py`): Three-region protection model:
- Protected head: system message, initial input, first model response, first tool call
- Compressible middle: from 2nd tool response onward
- Protected tail: final N turns (default 4)
Compression uses Gemini-3-Flash via OpenRouter. Supports batch mode with async parallel processing. Default target: 15,250 tokens with 750-token summary budget.

**Batch runner** (`batch_runner.py`): JSON checkpoint at `output_dir/checkpoint.json` tracking `completed_prompts`, `batch_stats`, `last_updated`. Smart resume via content matching (scans existing batch files, matches by prompt text rather than indices). Individual batch results saved incrementally to `batch_N.jsonl` for partial recovery. Final consolidation merges all batch files into `trajectories.jsonl`.

## 4. Checkpoint & Resumption

### 4.1 Batch Processing Checkpoints

The `batch_runner.py` system provides:
- JSON-based checkpoint files tracking completed prompt indices
- Content-based resume matching (robust against index mismatches between runs)
- Incremental per-batch JSONL writes (partial recovery on crash)
- Automatic consolidation of old + new batches on resume

**Known limitation** (identified in `tests/test_checkpoint_resumption.py`): Current implementation uses `pool.map()` which blocks until ALL batches complete — checkpoint saves only after all batches finish. Proposed fix: replace with `pool.imap_unordered()` + `multiprocessing.Lock()` for per-batch checkpoint saves.

### 4.2 Planned: Filesystem Checkpointing (TODO.md #8)

Not yet implemented. Planned features:
- Automatic snapshots after each agent loop iteration
- Git-based checkpoints in repositories (auto-commits to temporary branches)
- Non-git fallback: tar snapshots in `~/.hermes/checkpoints/`
- Rollback via `hermes rollback` CLI command

### 4.3 Context Compression Session Splits

When context compression triggers, the agent:
1. Flushes memory to disk
2. Creates a new session in SQLite with `parent_session_id` linking to the original
3. Injects todo snapshot after compression
4. Preserves protected turns + summary in the new session

## 5. What Hermes Does NOT Have

- **No autonomous model weight updates**: The RL training tool lets the agent orchestrate training runs, but the running agent instance does not update its own weights. It trains separate model checkpoints.
- **No feedback loops in the agent loop**: `environments/agent_loop.py` is explicitly a stateless execution engine — does not modify tool schemas or parameters based on outcomes, does not retain learning across loop instances.
- **No LICENSE file at standard path**: Despite declaring MIT in pyproject.toml, no LICENSE file exists at repo root. GitHub API returns `license: null`.
- **No plugin/extension system yet**: TODO.md describes a phased approach (enhanced hooks, plugin interface, MCP support) — not yet implemented.
- **No graph-based memory**: Memory is flat (two markdown files + SQLite session history). No knowledge graph, embedding store, or vector database is used by the core agent.

## 6. Architectural Patterns Worth Noting

**Frozen snapshot for cache preservation**: Memory is read once at session start and injected as a frozen snapshot. This prevents cache invalidation from mid-conversation memory updates while still persisting changes to disk immediately. Clever tradeoff between freshness and inference cost.

**Atomic file operations**: The memory system explicitly avoids `open("w")` + `flock` because `"w"` truncates before the lock is acquired. Instead uses temp-file + `os.rename()` for atomicity.

**Progressive skill disclosure**: Only skill names/descriptions are loaded into the system prompt. Full skill content is loaded on-demand via `skill_view()`, keeping the base prompt compact.

**Dual persistence with graceful degradation**: Session storage uses SQLite primarily but falls back to JSONL files when SQLite is unavailable.

**Content-based resume**: Batch checkpoint resumption matches on actual prompt text rather than indices, making it robust against dataset reordering.

**Three-process RL orchestration**: Training runs spawn API server, trainer, and environment server as separate processes with cascading startup waits and graceful termination in reverse order.

## 7. Key Relevance to OpenClaw

| Hermes Pattern | OpenClaw Relevance |
|---|---|
| Bounded markdown memory (MEMORY.md, USER.md) with frozen snapshots | Similar to our MEMORY.md / SOUL.md approach — validates the pattern |
| SQLite session store with FTS5 | We use ChromaDB for vector search; their FTS5 approach is simpler but less semantic |
| Skill generation as self-improvement | Analogous to our learnings/skills capture — their trigger heuristic (5+ tool calls) is concrete |
| Atomic file writes via temp+rename | Good practice we should verify in our own persistence |
| Context compression with protected regions | Relevant for our context management — their 85% threshold + 3-first/4-last protection is well-tested |
| RL training tool (Tinker-Atropos) | Significant: agent can orchestrate its own RL training. We should study whether this creates meaningful self-improvement loops |
| No graph memory, no embeddings in core | Confirms that simpler memory systems can work at production scale |
| Trajectory compression for training data | Useful for our own data generation pipeline if we pursue fine-tuning |
