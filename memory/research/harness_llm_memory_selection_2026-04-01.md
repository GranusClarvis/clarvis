# HARNESS_LLM_MEMORY_SELECTION — Research Note

The Claude harness’s `findRelevantMemories.ts` pattern—scan memory files, then use a cheaper LLM side-query to choose the top few memories—looks less like a replacement for embeddings and more like a precision layer on top of recall-oriented retrieval. The strongest external support comes from two-stage retrieval work: embedding search is fast and good at broad semantic recall, but it often admits noisy candidates; a second-stage reranker improves precision by selecting what actually matters for the current question. LlamaIndex describes this explicitly as high-recall first pass plus higher-precision reranking. Anthropic’s Contextual Retrieval work adds a useful warning: chunked/embedded retrieval frequently loses surrounding context, and contextualization plus reranking sharply reduces failed retrievals. A recent IR paper on LLM pre-filtering shows that narrowing the candidate set before full reranking can make smaller models competitive with larger ones, which is exactly the appeal of a Sonnet side-query over the whole memory store.

Implication for Clarvis: keep embeddings for scalable candidate generation, but insert an LLM rerank stage over a small candidate pool—ideally with memory-type metadata and short contextual summaries, not raw snippets alone. The likely best prototype is: embeddings/BM25 hybrid prefilter → compact contextual memory cards → LLM top-5 selection → inject only selected memories into heartbeat or task context. That should improve relevance while keeping token and latency budgets under control.
