## Night — Claude Code Debugging (Claude Code Opus session with Inverse)

### Problem Investigated
Clarvis reported Claude Code "hanging" when running Python scripts that import brain.py/chromadb.

### Root Cause Found
**Claude Code was NOT hanging — it was actively working.** Confirmed by reading Claude Code's own session logs:
- Session `96bd5ecb`: 33 entries of productive Opus work — read files, planned, edited code, was about to run tests when killed
- Other "hung" sessions had 19-37 entries of active work

Three issues combined to create the illusion of hanging:
1. **Output buffering**: `claude -p` produces NO stdout until task completes. "No new output" = still working.
2. **Opus is deliberate**: Each API round trip takes 15-30s. A 15-step task takes 5-10 minutes — normal.
3. **Short timeouts**: Clarvis used 30-180s timeouts — far too short for Opus multi-step tasks.
4. **Invalid `--cwd` flag**: Caused early invocations to fail with "unknown option" errors.
5. **`pty:true` as shell syntax**: OpenClaw exec prefix used as bash command — parse errors.

### Fixes Applied
1. **Claude Code skill rewritten** — added critical output buffering warning, proper timeout table (300-900s), removed `--cwd`, fixed all examples to use `cd /path && timeout N claude -p ...`
2. **AGENTS.md updated** — output buffering warning, correct invocation patterns, Opus as default model
3. **QUEUE.md commands fixed** — replaced `--cwd` with `cd /path &&` pattern
4. **11 orphaned Claude Code sessions cleaned up** — moved to /tmp/orphaned-cc-sessions/
