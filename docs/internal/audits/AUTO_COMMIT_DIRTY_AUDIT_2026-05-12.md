## (a) Script location + dirty-minute formula

**File:** `clarvis/cognition/obligations.py`

| Step | Location | Code |
|------|----------|------|
| dirty-minute compute | `obligations.py:289-299` | enumerates `git diff --name-only` + untracked files from `git status --porcelain`, takes the **minimum file `mtime`** across the first 20 files, computes `(time.time() - oldest_mtime) / 60` |
| commit message render | `obligations.py:470-475` | `f"chore: auto-commit {len(changed_files)} files (dirty {hygiene['dirty_age_minutes']:.0f}m)"` |
| trigger | `obligations.py:405-406` | gate: only auto-commits when `dirty_age_minutes >= 60` (i.e. dirty tree older than 1 hour) |
| stage step | `obligations.py:458-460` | `git add -A` (after secret + large-binary checks at lines 428-450) |

**Formula (exact):**

```python
# obligations.py:298-299
age_min = (time.time() - oldest_mtime) / 60
result["dirty_age_minutes"] = round(age_min, 1)
```

`oldest_mtime` = `min(os.path.getmtime(f) for f in changed_files[:20])`. This is the float-second mtime of the *oldest* file that `git` reports as dirty or untracked — not a per-commit elapsed-time clock. **The formula has no clamp**, so any file whose mtime is older than `time.time() - 60s` will report a `dirty_age_minutes` proportional to its on-disk age, regardless of when the file was actually changed in git's view.

## (b) Root cause of the `1096250m` outlier (commit `d679ff1`)

**Commit:** `d679ff1a24219dcc5648a3ffc4444f2e0800eac0` (2026-05-09 22:11:27 UTC).
**Files in commit:**

```
libatk1.0-0t64_2.52.0-1build1_amd64.deb   | Bin 0 → 55282 bytes
scripts/audit/p3_hyde_probe.py            | 547 ++++++++++++++++++++++++++++++++
```

**Diagnosis:** `1096250 minutes = 761.3 days ≈ 2.09 years`. Working backwards from the commit timestamp 2026-05-09 22:11 UTC:

```
2026-05-09 22:11 UTC  −  1,096,250 min  =  2024-04-07 17:10 UTC
```

The `.deb` package file `libatk1.0-0t64_2.52.0-1build1_amd64.deb` is a **Debian-built binary artifact whose tarball mtime is preserved as the build epoch** (`2024-04-07`). It was extracted/downloaded into the workspace and `git status --porcelain` reported it as untracked at `obligations.py:284-288`. Because the obligation tracker iterates the first 20 untracked + diffed files and takes the **minimum** mtime, this single 2-year-old `.deb` file dominated the calculation and produced the `1096250m` outlier.

**File:line responsible:** `clarvis/cognition/obligations.py:291-296` — the `oldest_mtime` reduction over `changed_files[:20]` is the bug surface. It correctly answers "what is the oldest mtime in my dirty set" but does not distinguish *content age* (legitimately edited code) from *artifact age* (extracted binaries with preserved upstream mtime).

**Not a clock skew or epoch overflow** — the system clock and `time.time()` are correct; the formula is mathematically correct; the input data is misleading.

## (d) Verdict — signal vs noise

**Verdict: 71% intended signal (state-file capture) + 25% safety-net legitimate (code captured when spawned agent neglected to commit) + 4% one-off outlier (the `.deb` artifact, fixed below). No noise to remove; one cosmetic bug to fix.**

The auto-commit class is **adding signal** to git history:
- It preserves the work that the heartbeat completed but failed to commit explicitly (catching a known agent failure mode where `partial_success` agents skip the final `git commit` step).
- It produces a clean audit trail of when each state file actually changed vs when it last got pushed (the dirty-minute is the latency between the change and the safety-net push).
- The 1 outlier (`d679ff1`) is a cosmetic display issue, not a behavioral bug — the commit itself was correct (it committed a real new file `scripts/audit/p3_hyde_probe.py`, plus the stray `.deb`).

**However** the `.deb` file represents a different problem: a 55KB Debian package was committed into a code repo. This is workspace pollution, not an auto-commit bug.

## (e) Recommendations

### Rec #1 — clamp `oldest_mtime` to commit-age, not file-age (cosmetic fix)

**Target:** `clarvis/cognition/obligations.py:289-299`

**Change:** filter the `changed_files[:20]` set to *tracked* files only when computing `oldest_mtime`. Untracked files (which can be just-extracted tarballs with preserved upstream mtimes) should not anchor the dirty-minute clock. Sketch:

```python
# Instead of iterating all changed_files (which includes untracked),
# use only files that git already tracks — those mtimes reflect *edit*
# events, not extraction events.
tracked_changed = subprocess.run(
    ["git", "diff", "--name-only"], ...
).stdout.strip().split("\n")
tracked_changed = [f for f in tracked_changed if f.strip()]
if tracked_changed:
    oldest_mtime = min(
        os.path.getmtime(os.path.join(WORKSPACE, f))
        for f in tracked_changed[:20]
        if os.path.exists(os.path.join(WORKSPACE, f))
    )
    age_min = (time.time() - oldest_mtime) / 60
    result["dirty_age_minutes"] = round(age_min, 1)
```

Untracked files would still be committed (the `git add -A` step is unchanged), but they would not anchor the `dirty Nm` display.

**Effect on history:** future outliers like `1096250m` disappear. The 60-minute threshold still triggers correctly because at least one tracked file exists in every legitimate auto-commit scenario.

### Rec #2 — add `.deb`, `.tar.gz`, `.zip` to a workspace-root `.gitignore` allow-list (workspace hygiene)

**Target:** `.gitignore`

**Change:** explicitly ignore Debian/RPM packages + tarball archives at workspace root, since these are install-time artifacts that should never enter the repo:

```
# Install-time artifacts (preserve upstream mtimes, pollute history)
*.deb
*.rpm
*.tar.gz
*.tgz
*.zip
!skills/**/*.zip   # exception: skill bundles are legitimate
```

**Effect:** the `libatk1.0-0t64_*.deb` would have been silently ignored at `git status` time, and the `d679ff1` commit would have shipped only `p3_hyde_probe.py`. Cleaner history, no behavior change to the auto-commit logic.

### Rec #3 — narrow auto-commit scope, NOT recommended

The original task body mentions "narrow scope to specific paths" as a possible recommendation. **Audit verdict: do not adopt this.** The state-file pattern is the entire point of auto-commit (capturing QUEUE.md + status.json drift); restricting scope to those two files would mean code drift from neglected agents never gets caught. The current `git add -A` + secret/binary guards is the right scope.
